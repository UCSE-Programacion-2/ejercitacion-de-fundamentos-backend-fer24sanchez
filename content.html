Introducción a FileSystem API

Si bien hemos tenido una aproximación a lo que este módulo brinda dentro de Node.js, FileSystem API es una herramienta mucho más completa y efectiva cuando se trata de manipular, tanto archivos, como también carpetas o directorios.

Profundicemos un poco más en sus características desarrollando ejemplos prácticos, con todo lo que este módulo nos ofrece.

Qué es FileSystem API

Refresquemos conceptos:


El módulo fs integrado en Node.js proporciona una API para interactuar con el sistema de archivos en el servidor. Dentro de todas sus prestaciones, podemos destacar la creación, escritura, modificación y eliminación de archivos dentro de un S.O.

Nuestras pruebas y ejemplos se enfocarán íntegramente dentro de un proyecto Node.js.

Para integrar FileSystem API dentro de un proyecto Node.js, debemos declarar el módulo fs utilizando la función JS require(). De esta forma ya queda disponible para que lo utilicemos dentro de nuestros proyectos.

Una vez declarado, este módulo cuenta con un montón de métodos JS, los cuales nos facilitan realizar todo tipo de interacción sobre un archivo, o varios, integrados mayormente en nuestro proyecto.


Veamos a continuación cuáles son los métodos principales para manipular las diferentes operaciones sobre archivos:

Para que scomprendan la seriedad de manejar archivos, tengan en cuenta estos detalles técnicos:

  1. Sobreescritura Fatal: writeFile es destructivo. Si el archivo ya existe, borra todo lo anterior y pone el contenido nuevo. Es como hacer un format a esa celda de datos.

  2. El formato UTF-8: Siempre especifiquen la codificación. Como ingenieros, debemos garantizar que los caracteres especiales (tildes, eñes) se guarden y lean correctamente en cualquier sistema operativo.

  3. Persistencia de Objetos: En el Backend, casi nunca guardamos texto plano. Guardamos JSON.
    Para esto, debemos convertir el objeto a string antes de escribirlo:

const alumno = { nombre: "Fabio", legajo: 1234 };

await fs.writeFile('alumno.json', JSON.stringify(alumno, null, 2));

Validar que el archivo exista previamente:

Como Ingenieros Informáticos, sabemos que uno de los errores más comunes y evitables es intentar leer o modificar un archivo que no existe, lo que provocaría que nuestro servidor "explote" (crush).

En las versiones modernas de Node.js, para verificar la existencia de un archivo de manera profesional, usamos el método access del módulo fs/promises

Validación de Existencia con fs.access

Aunque en versiones muy antiguas existía fs.exists, hoy el estándar de ingeniería es usar access. Este método comprueba si el archivo es accesible según los permisos que le pidamos (por defecto, si existe).

👨‍🏫 Notas de Ingeniería

Para que los entiendan por qué usamos access y no otras funciones:

  1. Manejo de Excepciones: En el mundo de las Promesas, la ausencia de un archivo se trata como un error (catch). Es una forma muy segura de programar.

  2. Permisos: access no solo mira si el archivo está ahí, también puede chequear si tenemos permiso de lectura o escritura (fs.constants.R_OK o W_OK).

  3. Principio de Robustez: Antes de hacer un readFile o un appendFile, siempre es buena práctica validar la ruta. Esto evita que el proceso de Node.js se detenga inesperadamente.

Leer un archivo

El ciclo de la persistencia se completa cuando podemos recuperar lo que guardamos. Leer un archivo es una operación asíncrona crítica: el servidor no puede detenerse a esperar que el disco rígido responda; debe seguir atendiendo otras peticiones mientras los datos viajan a la memoria RAM.


Para un Ingeniero Informático, el método fs.readFile es la herramienta clave para cargar configuraciones, bases de datos en JSON o plantillas de texto.


Al igual que con los métodos anteriores, usamos la versión de Promesas para mantener un código limpio y profesional.


Cuando leemos contenido de un archivo en un servidor web, este contenido se lee en forma de stream. Por ello, debemos indicar la codificación de caracteres del archivo leído, para así evitar la generación de un buffer, en pos de poder visualizar correctamente el texto almacenado en éste.

El parámetro “utf-8” será quien acomode el contenido en formato stream, y haga que este sea legible en la codificación de caracteres en la cual fue escrita.

👨‍🏫 Notas de Ingeniería

Para que no cometan los errores clásicos de principiante:

  1. ¿Buffer o String?: Si olvidan el segundo parámetro 'utf-8', Node.js les devolverá un Buffer (datos hexadecimales). Como humanos y desarrolladores web, casi siempre necesitamos el formato de texto.

  2. El poder de JSON.parse(): Los archivos guardados en el disco son siempre cadenas de texto. Para poder usar métodos de arreglos como .map() o .filter() sobre esos datos, es obligatorio convertirlos primero con JSON.parse().

  3. Tratamiento de Errores: Si el archivo está vacío o el JSON está mal formado (le falta una coma, por ejemplo), JSON.parse() hará que el servidor explote. Por eso, el bloque try-catch es obligatorio aquí.

Archivos JSON

Un archivo JSON es un formato de texto utilizado para almacenar y transmitir datos estructurados.

El nombre JSON es una sigla proveniente de JavaScript Object Notation, aunque se utiliza en muchos otros lenguajes de programación además de JavaScript.


Un archivo JSON está formado por una colección de pares de clave/valor. 


Los datos se almacenan en un formato similar al de un objeto JavaScript, con propiedades y valores separados por dos puntos (:), y cada par clave/valor separado por una coma (,). 


Los datos también pueden estar anidados, permitiendo la creación de estructuras de datos complejas.

Array de Objetos

Archivo JSON

En el formato JSON, se utiliza la misma estructura de objetos y arrays que en JavaScript, pero con la diferencia de que las claves y los valores se escriben entre comillas dobles y que no se permite el uso de comillas simples. 


Además, el último elemento de un objeto o array no debe tener una coma después de él.

¿Para qué suelen utilizarse estos?

Los archivos JSON son ampliamente utilizados en aplicaciones web para intercambiar datos entre el servidor y el cliente, ya que son fáciles de leer y escribir, y son compatibles con la mayoría de los lenguajes de programación.

¿Podemos editar archivos JSON desde Node.js?

Es posible su edición, utilizando el módulo FileSystem de Node.js. Debemos trabajarlo combinando su lectura, conversión al formato de array de objetos, y luego de agregar o modificar sus elementos, volver a escribir el contenido actualizado en el archivo en cuestión.

Escribir contenido en un archivo

Existe la posibilidad de ir agregando contenido de forma parcial dentro de un archivo ya creado. 


Para ello, el método .appendFile() es el apropiado.

A diferencia de writeFile, que es destructivo y borra todo lo anterior, appendFile añade el nuevo contenido al final del archivo. Es la herramienta por excelencia para cualquier Ingeniero Informático que necesite crear sistemas de logs, historiales de chat o registros de auditoría.

Como todos los métodos de FileSystem API, es simple de utilizar. 

El método .appendFile() verifica si el archivo existe:

  • Si existe: pega el nuevo contenido al final.

  • Si no existe: lo crea (igual que writeFile).

Ejecutada nuestra aplicación Node.js, ya podemos validar a continuación el archivo .TXT creado, que el mismo contiene el texto agregado con esta última función JS.


Al ejecutar la función .appendFile(), pudimos comprobar que el contenido agregado al archivo .TXT se grabó de forma automática. El método en cuestión, además de agregar contenido en un archivo también lo guarda, evitando así que tengamos que recurrir a una segunda función para esto último.

👨‍🏫 Notas de Ingeniería (Análisis de Casos)

Como profesional, debés saber cuándo elegir un método sobre otro:

  1. Logs de Servidor: Es el uso más común. Cada vez que hay un error o una petición en Express, usamos appendFile para que el historial crezca sin perder lo que pasó ayer.

  2. El carácter de Nueva Línea (\n): Es vital. Si no agregamos \n al final de nuestro string, la próxima vez que hagamos un append, el texto se pegará justo al lado del anterior en la misma línea, creando un desorden ilegible.

  3. ¿JSON con appendFile?: ¡Cuidado aquí! No se puede usar appendFile directamente para agregar un objeto a un array dentro de un archivo .json. Si lo hacés, romperías el formato del JSON. Para eso, la técnica profesional es: Leer -> Parsear -> Push -> Escribir todo de nuevo.

Borrar un archivo

Nos queda ver cómo podemos eliminar un archivo existente utilizando FileSystem API.


Para ello, el método .unlink() es la herramienta que nos ayudará a realizar esta tarea.

Como bien dijimos, el método es fácil de utilizar, recibiendo como primer parámetro el nombre del archivo en cuestión, y controlamos cualquier posible error, o la eliminación efectiva del archivo en cuestión.

👨‍🏫 Notas de Ingeniería (Análisis de Seguridad)

Para sus proyectos y clases, tengan en cuenta estos puntos críticos de seguridad y rendimiento:

  1. Operación Irreversible: A diferencia de cuando borramos algo en Windows o macOS, fs.unlink no envía el archivo a la papelera. Se borra directamente. Por eso, siempre debemos estar 100% seguros antes de ejecutarlo.

  2. Validación Previa: Aunque podemos manejar el error con un try-catch, muchas veces es más elegante usar el fs.access que vimos antes para verificar si el archivo existe antes de intentar borrarlo.

  3. Permisos de Sistema: En servidores reales (Linux/Unix), el usuario que ejecuta Node.js debe tener permisos de escritura sobre la carpeta para poder eliminar archivos de ella.

El mismo módulo FileSystem nos permite trabajar con carpetas o directorios del servidor. Como vimos al inicio de esta clase, existen métodos que nos permiten crear, leer y actualizar directorios, como también eliminarlos. Veamos entonces cuáles son y cómo se implementan.

👨‍🏫 Notas de Ingeniería (Tips de Arquitectura)

Para que entiendan la importancia de este método en un entorno profesional:

  1. La importancia de recursive: Por defecto, si intentás crear ./data/usuarios y la carpeta data no existe, Node te dará un error. Con { recursive: true }, Node se encarga de crear todo lo que falte en el camino. Es la forma más robusta de programar.

  2. Validación de Existencia: Al igual que con los archivos, a veces es útil usar access para saber si la carpeta ya existe. Sin embargo, mkdir con recursive: true no da error si la carpeta ya está ahí; simplemente no hace nada. ¡Es muy cómodo!

  3. Orden en el Backend: En sus proyectos con Express, usarán esto para organizar:

    • ./uploads: Para fotos de perfil o archivos subidos.

    • ./temp: Para archivos procesados temporalmente.

    • ./logs: Para los archivos de registro que vimos con appendFile.

El método .readdir() permite realizar una lectura del contenido especificado.

Renombrar el contenido de un directorio/carpeta:

El método .rename() cambia el nombre del directorio especificado como parámetro.


Si ocurre un error al realizar este cambio, se muestra un mensaje en la consola.

Tip: Si nuevaRuta está en el mismo directorio, simplemente cambia el nombre. Si está en otro, lo mueve. ¡Es 2 por 1!

Eliminar el contenido de un directorio/carpeta:

El método .rmdir() elimina el nombre de un directorio del sistema de archivos del servidor.


Este método se usa específicamente para borrar carpetas. Al igual que con mkdir, tenemos un parámetro clave llamado recursive.

  • Sin recursive: Solo borra la carpeta si está vacía. Si tiene archivos, da error (es más seguro).

  • Con recursive: true: Borra la carpeta y todo lo que tenga adentro (archivos, subcarpetas). ¡Cuidado con este!

👨‍🏫 Notas de Ingeniería 

Para que tno tengan accidentes con los datos en el servidor:

  1. Diferencia entre unlink y rmdir:

    • unlink es para archivos.

    • rmdir es para directorios. Si intentan usar uno con el otro, Node les devolverá un error de sistema.

  2. Riesgo de recursive: true: Como ingenieros, debemos usar el borrado recursivo con extrema precaución. Es una operación atómica que no tiene vuelta atrás. Siempre es bueno validar con un if antes de ejecutarlo.

  3. Modernidad: En versiones muy nuevas de Node.js, también existe fs.rm(), que es un método más genérico que puede borrar tanto archivos como carpetas, pero rmdir sigue siendo el estándar académico para directorios.

Analogía con comandos de Bash

Muchos de estos métodos respetan el nombre de los comandos dentro de los File System tipo Linux - Mac. 


Si tienes experiencia en los mismos, serán fáciles de recordar, sino, aprender estos métodos te ayudarán mucho a luego poder manejar esta estructura de directorios sin problema.

Y, si utilizas Windows, los nombres de los comandos son algo diferentes en la Command Line, pero puedes recurrir a instalar PowerShell, una aplicación de Microsoft que cambia la estructura de archivos y directorios de Windows a través de una línea de comandos diferente, haciéndola parecida a la interfaz Linux.


Y lo mejor de todo, es que podrás unificar los comandos para manejar archivos y carpetas en Windows, tal como lo harías con Linux ó Mac.

Son simples de utilizar pero, en determinadas situaciones, requieren de cuidados particulares porque la reescritura de contenido o eliminación de un archivo o carpeta, debe ser controlada por la lógica de nuestro código.

El módulo Path

El módulo path de Node.js proporciona un conjunto de métodos que permiten trabajar con rutas de archivos y directorios, de forma independiente al sistema operativo en el cual se ejecuta Node.js. 


En particular, este módulo permite trabajar con rutas en formato de cadena y obtener información sobre las mismas, como el nombre del directorio, el nombre del archivo, la extensión del archivo, entre otros.

En el ecosistema de Node.js, trabajar con rutas de archivos puede ser un dolor de cabeza si no se hace de forma profesional, especialmente cuando el código debe correr en distintos sistemas operativos.

Aquí es donde entra el módulo path. Como Ingeniero Informático, sabés que Windows usa barras invertidas (\) y Linux/macOS usan barras inclinadas (/). El módulo path resuelve esto automáticamente, garantizando que nuestro software sea multiplataforma.

Métodos esenciales del módulo path

Para trabajar con Express, hay cuatro herramientas que vas a usar el 90% del tiempo:

1. path.join(...segments): Es el método más importante. Une varios fragmentos de una ruta usando el separador correcto del sistema operativo donde se esté ejecutando.

2. path.extname(path): Devuelve la extensión del archivo. Es vital cuando querés validar si un usuario está subiendo un archivo permitido (por ejemplo, solo .png o .jpg).

3.  path.basename(path): Extrae el nombre del archivo de una ruta completa. Muy útil para mostrar nombres en una lista de descargas.

4. __dirname y __filename: En el Node.js moderno (usando import), las variables globales __dirname y __filename no existen por defecto. Como ingenieros, debemos "reconstruirlas" para saber siempre en qué carpeta estamos parados. 

👨‍🏫 Notas de Ingeniería

Recordá estos puntos clave:

  • Evitar la concatenación manual: Nunca hay que hacer 'carpeta/' + 'archivo.js'. Siempre usá path.join(). La concatenación manual es la fuente número uno de errores cuando desplegás tu servidor en la nube.

  • Normalización: Si un usuario envía una ruta extraña como ../../config.json, podés usar path.normalize() para limpiar la ruta y evitar problemas de seguridad (Directory Traversal).

  • Rutas Absolutas vs Relativas: Para configurar lo siguiente que veremos que es express.static, Express exige una ruta absoluta. Por eso la combinación de __dirname y path.join es el estándar.

Servir un sitio web con Express JS

Express JS tiene un montón de ventajas respecto al módulo HTTP. Una de ellas es cuando debemos servir un sitio web frontend.

Debemos vincular el módulo filepath, fileSystem API, la variable global __dirname y a su vez controlar el tipo de recurso asociado al documento HTML para servirlo correctamente (CSS, JS, imágenes, etcétera)

__dirname es una variable la cual nos devuelve una ruta completa de carpetas y subcarpetas, partiendo como base de la raíz del disco del sistema operativo donde Node.js está ejecutándose.


Es parte de las variables de entorno de Node.js, algo que estudiaremos con mayor detalle en los próximos encuentros.



Tengamos presente que la estructura de carpetas que devuelve __dirname, varía ligeramente de acuerdo al S.O.



Para resolver esta tarea, debemos referenciar al módulo path, luego usamos el método static() de Express JS. Por último, invocamos al método path.join() concatenando __dirname junto a la subcarpeta que contiene el sitio web.

Efectivamente, Express JS, es el Framework para crear servidores web que nos facilita la vida y acorta notoriamente los tiempos de desarrollo web.


👇 Aquí, un claro ejemplo de ello.

Express JS es una herramienta más potente y completa para construir aplicaciones web y APIs que el módulo HTTP. 


Si bien el módulo HTTP es útil para situaciones simples, como servir archivos estáticos, Express JS proporciona una funcionalidad más avanzada y un conjunto de herramientas más completo para manejar aplicaciones web y APIs de manera más sofisticada.

👨‍🏫 Notas de Ingeniería

  1. Seguridad: Solo los archivos dentro de la carpeta que definas en express.static serán visibles para el mundo. El resto de tu código (como el package.json o tus variables de entorno) queda protegido.

  2. El archivo index.html: Por convención, si alguien entra a la raíz (/), Express buscará automáticamente un archivo llamado index.html dentro de la carpeta pública y lo servirá. No hace falta crear una ruta app.get('/') manualmente.

  3. Rutas Absolutas: Siempre debén usar path.join(__dirname, ...) o métodos similares. Esto garantiza que tu servidor funcione igual de bien en tu PC, en un servidor Linux o en la nube (AWS/Azure), sin importar desde qué carpeta se ejecute el proceso.

 Su nombre es, literalmente, ‘.env’.

Su estructura es, básicamente, un archivo de texto plano que se encuentra en la raíz de tu proyecto Node.js y contiene las variables de entorno que necesita tu aplicación. 

Cada variable de entorno se define como una línea en el archivo en el formato NOMBRE=VALOR, donde NOMBRE hace referencia a la variable en sí y, VALOR, al valor almacenado en ésta.

Aquí tenemos un ejemplo de cómo referir cada una de las variables creadas dentro del archivo .env, y los valores asociados a éstas. Así, variables claves como PORT, URL de bases de datos, y subcarpetas con aplicaciones frontend, no estarán atadas a una URL estática dentro del código de nuestras aplicaciones backend.

Implementar las variables de entorno

Creado el archivo .env y declaradas las variables pertinentes, nos queda instalar la dependencia dotenv con npm, y luego referenciarla en nuestra aplicación. Finalmente, el método .config() agrega las variables de entorno definidas en este archivo, dentro del objeto JS process.env.

A través del objeto process.env.’ENV_VARIABLE’, accederemos al valor definido en esta variable. Ahora, el servidor web, escuchará en el puerto definido PORT (variable de entorno). Si no existe dicha variable, entonces asumirá el puerto 3000 como predeterminado.

Con las variables de entorno PORT y WEBSITE_FOLDER, ya evitamos tener que forzar a la aplicación web a trabajar en un puerto específico o ubicar al sitio web en una subcarpeta determinada.


Cada entorno involucrado en las etapas de desarrollo y pruebas, solo deberá cambiar los valores de las variables de entorno, por las rutas específicas de su entorno de trabajo.

La notación punto (.) antes de la palabra env, lo convierte en un archivo invisible. 


Podremos verlo a través de un editor de código como VS Code, pero no será visible en una carpeta o subcarpeta de servidor, sobre todo si el mismo es un servidor Linux o derivado de sistemas Unix.

Regla de Oro: ¡Nunca subas el archivo .env a GitHub! Agrégalo siempre a tu .gitignore.

Leer archivos .env sin dependencias

Carga Programática (process.loadEnvFile())

Si preferís tener el control dentro del código (similar a como hacíamos con dotenv.config()), Node.js incorporó este método en el objeto global process.


Nota técnica: Si el archivo se llama distinto o está en otra carpeta, podés pasarle la ruta: process.loadEnvFile('./config/.env.dev').

Carga mediante Flag de Terminal (--env-file)

Esta es la forma más directa y recomendada para entornos de desarrollo y producción. Le indicamos a Node.js qué archivo de configuración debe cargar antes de ejecutar nuestro código.

Ventajas de esta forma:

  • Carga Temprana: Las variables están disponibles desde el milisegundo cero, incluso antes de que se importe cualquier otro módulo.

  • Sin código extra: No necesitas escribir ni una sola línea de importación o configuración dentro de tus archivos .js.

  • Versatilidad: Podés cambiar de entorno fácilmente disparando diferentes archivos: node --env-file=.env.production index.js

👨‍🏫 Notas de Ingeniería 

Diferencias clave para que decidan cuál usar:

  • Librerías vs Nativo: dotenv sigue siendo útil si necesitás funciones avanzadas (como expansión de variables), pero para el 90% de los casos, lo nativo es más eficiente y liviano.

  • Compatibilidad: Recordá que estas funciones requieren Node.js 20.6.0 o superior. Si el servidor donde vas a desplegar es más viejo, tendrás que seguir usando dotenv.

  • Prioridad: Si usás el flag --env-file y también llamás a process.loadEnvFile(), Node gestionará las variables, pero generalmente la carga por terminal es la más limpia para la arquitectura del sistema.
